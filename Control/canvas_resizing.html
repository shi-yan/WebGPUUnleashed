3.0 Canvas Resizing

At this point, all of our samples are rendered in a fix sized canvas. But for real applications, we often want the canvas to fully occupy the full page or its container. This is a bit tricker than you might expect to resize a canvas. In this tutorial, let's look at how to achieve it.

Every canvas has two sizes, one is the framebuffer size, this size is the same as the rendering attachment. let's term it the rendering size. The other size is used for presenting the canvas onto the webpage. let's call it the display size. Our rendering applies to the framebuffer and once rendered, we can treat it as an image of the rendering size. Then we put this image onto the webpage using the display size. If the two sizes don't match, the rendered result will be stretched to match the display size.

The rendering size can be defined by either specifying the width and height attributes through the html tag:

<canvas width="640" height="480"></canvas>

or it can be configured in javascript:

canvas.width = 640;
canvas.height = 480;

The display size can be controlled by CSS:

<canvas style="width: 400px; height: 300px;"> </canvas>

Or you can specify the canvas to have a size of 100% of its parent so that it can be maximized. However, this will only affect the display size, but not the rendering size. it's up to our developers to match the two sizes. To update the rendering size, we first need to capture the canvas resizing event. We can do this using the ResizeObserver.

        let timeId = null;
        const resizeObserver = new ResizeObserver((entries) => {
            if (timeId) {
                clearTimeout(timeId);
            }
            timeId = setTimeout(() => {
                requestAnimationFrame(render);
            }, 100);
        });
        ...
        resizeObserver.observe(canvas);

the ResizeObserver can let you specify a call back function. When the canvas has been resized, this callback will be called. The callback function should trigger the rendering of a new frame, in the new frame, we should adjust the rendering size, and update the framebuffer and the depth buffer accordingly.

A naive approach would call requestAnimationFrame directly, but we don't want to call this function too frequently. Consider for example, when we resize the window, this will trigger a series of resizing events. Immediately rerender the scene maybe unnecessary as the window size won't be settled until we have stopped resizing the window. Also, since rendering might be costly, frequent calling it during resizing might make resizing feels sluggish. here in the above code snippet, we want to throttle the rerendering by using a timer. The ResizeObserver's callback will apply a 100ms delay to requestAnimationFrame, and if another resizing event is triggered before the timeout fires, we reset the timeout. The way we can batch the execution of rerendering.

        async function render() {
            const devicePixelRatio = window.devicePixelRatio || 1;
            let currentCanvasWidth = canvas.clientWidth * devicePixelRatio;
            let currentCanvasHeight = canvas.clientHeight * devicePixelRatio;

            if (depthTexture === null || currentCanvasWidth != canvas.width || currentCanvasHeight != canvas.height) {
                canvas.width = currentCanvasWidth;
                canvas.height = currentCanvasHeight;

in the rendering function, we first obtain the pixel density via window.devicePixelRatio , on high dpi screen, such as that of a macbook, this value can be two, meaning we can put 4x more pixels in the same area of a normal pixel. Hence we can create a framebuffer 4x the size of the canvas to make our rendering finer. On most conventional monitors, this value is simply 1.0. In this scenario, we simply let the framebuffer size match that of the canvas.

If we detect the canvas has changed, i.e. the old canvas.width doesn't match the new size, we need to update a series of resources. The first to update the the depth texture. Our color rendering target is maintained automatically by the canvas, but the depth is maintained by us. Hence when the canvas size has changed, we have to release the old depth map and recreate a new one with the right size. The second thing to update is the projection matrix, as we recall, its creation is related to the aspect ratio of the canvas. Since we can't directly update the uniform buffer, because a uniform buffer can't be a MAP_WRITE at the same time, we will have to put the matrix in a staging buffer first and then do a buffer copy.
