3.4 Using Web Workers

WebGPU can be executed in a worker to further improve performance. In this example, let's see how to do that. The concept is simple, what we need to do is create a worker script, and we turn the canvas into an offscreen canvas and send it to the worker. The worker basically carry out the rendering as what we did before. This way the entire rendering work load is delegated to the worker and hour main thread can work on something else in parallel.

First, let's examine how to implement the worker script. the general structure of the worker script is as the follows:

self.addEventListener('message', (ev) => {
    switch (ev.data.type) {
        case 'run': {
            try {
                run(ev.data.offscreenCanvas, ev.data.code);
            } catch (err) {
                self.postMessage({
                    type: 'log',
                    message: `Error while initializing WebGPU in worker process: ${err.message}`,
                });
            }
            break;
        }
    }
});

async run(canvas, code) {...}

we need to have an event listener accepting the run message from the main thread. This event listener serves as the entry point of our rendering process. when we have received a message from the main thread, we expect three pieces of data: a message type, the shader code and the offscreen canvas. all of them are passed in as the message payload.

upon receiving the message, we first check the message type. Then we pass both the offscreen canvas and shader code into the run function. The run function is exactly the same as the previous rendering function, we won't look into its details.

now, let's focus on the main thread, and see how do we message the worker to start rendering.

    const worker = new Worker('./worker.js');
    ...
    try {
        const offscreenCanvas = canvas.transferControlToOffscreen();
        const devicePixelRatio = window.devicePixelRatio || 1;
        offscreenCanvas.width = canvas.clientWidth * devicePixelRatio;
        offscreenCanvas.height = canvas.clientHeight * devicePixelRatio;
        let code = document.getElementById('teapot_shader').innerText;
        worker.postMessage({ type: 'run', offscreenCanvas, code }, [offscreenCanvas]);
    } catch (err) {
        console.warn(err.message);
        worker.terminate();
    }

here we first load the worker script. Then we call transferControlToOffscreen to turn the canvas to an offscreen canvas and we scale its size by devicePixelRatio. After that, we obtain the shader code as a string. Finally, we post a run message to the worker to trigger the rendering. notice that when post the message, not only we need to put the offscreen canvas in the message payload, we also need to put it in the sequence of transferable.

a bit word about the concept of transferable object in javascript. an object being transferable means that there are some resources behind this object such that it's unsafe to be accessed simultaneously by more than one thread. Hence the safe way to do is transferring the object's ownership to the worker thread. We need to request this transferation by explicitly put this object in the array of transferable array (the second parameter of the postMessage call).

That's all we need to use worker for rendering.
