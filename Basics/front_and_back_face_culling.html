1.11 Front and Back Face Culling

In this section, we explore the concepts of front and back faces, a crucial aspect of rendering. You may have noticed the cullMode field in the pipeline descriptor's primitive settings, where we've set it to back or none. But what does cullMode mean, and why is culling significant?

Culling involves the removal of unnecessary geometries that aren't currently visible. These geometries might lie outside the current view or face away from the observer. In the primitive settings, cullMode specifically pertains to the "face culling mode". Setting it to none means we want to preserve all triangles, while back instructs the GPU to discard back-facing triangles.

Inside the shader code, once each triangle is rasterized into fragments, we can determine if a fragment is part of a triangle facing toward or away from us. Leveraging this information, we can assign different colors to fragments based on their orientation, allowing us to render outside and inside surfaces with distinct colors.

While these concepts might seem straightforward, they serve as a foundation for understanding depth, or the Z-buffer, which we'll explore in the next chapter. Visualizing front and back faces differently helps us grasp the artifacts that can arise when depth testing is disabled.

@fragment
fn fs_main(in: VertexOutput, @builtin(front_facing) face: bool) -> @location(0) vec4<f32> {
    if (face) {
        return vec4<f32>(0.0, 0.0, 1.0, 1.0); // Blue for front-facing
    } else {
        return vec4<f32>(0.0, 1.0, 0.0, 1.0); // Green for back-facing
    }
}

In the JavaScript code, all we need to do is ensure that back face culling is disabled by setting the cullMode to none in the primitive section of the pipeline descriptor.

const pipelineDesc = {
    ...
    primitive: {
        topology: 'triangle-strip',
        frontFace: 'ccw',
        cullMode: 'none'
        }
    };

However, you may notice that the rendering doesn't appear entirely accurate, with some faces that should be occluded by others penetrating through. This occurs because we haven't yet enabled depth in our setup. Without depth, we can't determine which triangles should occlude others. Drawing order is determined mainly by the order in the position buffer, not by triangles' distance from the camera. We'll address this issue in the next tutorial.
