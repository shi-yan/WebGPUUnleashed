1.4 Using Different Vertex Colors

In this new tutorial, we're making slight modifications to the previous example. Instead of hard-coding colors in the shader, we'll pass vertex colors as data into the shader.

Let's begin by examining the changes on the shader side. In addition to the position attribute at @location(0), we've introduced another input parameter at @location(1), which we've named inColor. The format for inColor is a vector of three floats representing R, G, and B. In the vertex stage, where previously we hardcoded the out.color field, we now simply copy the input inColor to it. The rest of the code remains the same. With this example, we aim to demonstrate how to pass multiple attributes, beyond just vertex positions, using different locations.

By assigning different colors to different vertices, we aim to address the question we raised earlier: how are fragment colors generated for those fragments that are located in the middle of a triangle?

<script id="shader" type="wgsl">
// Vertex shader

struct VertexOutput {
    @builtin(position) clip_position: vec4<f32>,
    @location(0) color: vec3<f32>,
};

@vertex
fn vs_main(
    @location(0) inPos: vec3<f32>,
    @location(1) inColor: vec3<f32>
) -> VertexOutput {
    var out: VertexOutput;
    out.clip_position = vec4<f32>(inPos, 1.0);
    out.color = inColor;
    return out;
}

// Fragment shader

@fragment
fn fs_main(in: VertexOutput) -> @location(0) vec4<f32> {
    return vec4<f32>(in.color, 1.0);
}

</script>

Now, let's explore how to set up the pipeline. The new pipeline is quite similar to the previous one, with the key difference being that we now need to create a color buffer containing colors for all the vertices and feed it into our pipeline.

The steps involved are quite similar to how we handled the position buffer. First, we need to create a color attribute descriptor. Note that we set shaderLocation to 1, corresponding to the inColor attribute at @location(1). The format for the color attribute remains a vector of three floats.

Next, we create the buffer layout descriptor, which informs the pipeline how to break down the color buffer for each vertex. We assign the color attribute descriptor to the attributes field. The arrayStride is still 4 * 3 because a float occupies 4 bytes, and there are 3 floats for each color. The stepMode is set to vertex because each vertex will have one color.

After defining the RGB data in CPU memory by creating a Float32Array, with the first vertex being red, the second green, and the third blue, we proceed to create a GPU buffer and copy the data onto the GPU.

Here's how it's done step by step:

  1. We first define a buffer descriptor by specifying the size of the buffer and setting the usage flag as vertex since we'll use the color attribute at the vertex stage.

  2. We specify mappedAtCreation for data copying.

  3. Then, we create the buffer, and with the mapped buffer range, we create a mirrored buffer in CPU memory.

  4. We copy the data into the mapped buffer.

  5. Finally, we unmap the buffer.

At this point, we've successfully duplicated the same color values on the GPU.

<script>
    async function webgpu() {
        ...

        const colorAttribDesc = {
            shaderLocation: 1, // @location(1)
            offset: 0,
            format: 'float32x3'
        };

        const colorBufferLayoutDesc = {
            attributes: [colorAttribDesc],
            arrayStride: 4 * 3, // sizeof(float) * 3
            stepMode: 'vertex'
        };

        const colors = new Float32Array([
            1.0,
            0.0,
            0.0, // red
            0.0,
            1.0,
            0.0, // green
            0.0,
            0.0,
            1.0 // blue
        ]);

        const colorBufferDesc = {
            size: colors.byteLength,
            usage: GPUBufferUsage.VERTEX,
            mappedAtCreation: true
        };

        let colorBuffer = device.createBuffer(colorBufferDesc);
        const writeArrayColor = new Float32Array(colorBuffer.getMappedRange());
        writeArrayColor.set(colors);
        colorBuffer.unmap();

    }

    webgpu();
</script>

After this step, we define the pipeline descriptor. The only difference than before is we add the colorBufferLayoutDescriptor in the buffers list.

When we create the command, we also need to set vertex buffer 1 with the color buffer. The rest remains pretty much the same.

<script>

    async function webgpu() {
        ...

        const pipelineDesc = {
            layout,
            vertex: {
                module: shaderModule,
                entryPoint: 'vs_main',
                buffers: [positionBufferLayoutDesc, colorBufferLayoutDesc]
            },
            fragment: {
                module: shaderModule,
                entryPoint: 'fs_main',
                targets: [colorState]
            },
            primitive: {
                topology: 'triangle-list',
                frontFace: 'ccw',
                cullMode: 'back'
            }
        };

        ...

        passEncoder = commandEncoder.beginRenderPass(renderPassDesc);
        passEncoder.setViewport(0, 0, canvas.width, canvas.height, 0, 1);
        passEncoder.setPipeline(pipeline);
        passEncoder.setVertexBuffer(0, positionBuffer);
        passEncoder.setVertexBuffer(1, colorBuffer);
        passEncoder.draw(3, 1);
        passEncoder.end();

        device.queue.submit([commandEncoder.finish()]);
    }

    webgpu();
</script>

Now, let's observe what happens when we run this code. As you can see, we're presented with a colorful triangle. Each vertex is rendered with the provided color, displaying red, green, and blue. However, in the middle of the triangle, a color transition occurs.

This automatic color transition is created by the GPU, a process we refer to as interpolation. In fact, it's not limited to colors; any value we output at the vertex stage will be interpolated by the GPU to assign appropriate values for every fragment, particularly for those that are not located at the vertices.

The value interpolation calculated for a fragment's is based on its distance to the vertices, following a bilinear scheme. This mechanism is highly convenient because, considering there are far more fragments than triangles, we can't specify values for all fragments individually. Instead, we rely on the GPU to generate these values sparsely based on the values defined only at each vertex.
