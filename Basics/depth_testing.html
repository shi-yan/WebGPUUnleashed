1.12 Depth Testing

In this section, we'll dive into the concept of depth testing, a crucial technique for resolving issues like the inside surfaces appearing inverted. Depth testing allows us to determine the rendering order of fragments from front to back, ensuring that we only draw the fragments closest to the camera while discarding those behind them.

In camera coordinates, which include the x and y axes for the screen plane and the z-axis for depth, the z-value represents the depth of each fragment. Sorting fragments based on their z-values becomes essential for rendering.

To accomplish this, we introduce an additional temporary buffer known as the depth buffer or z-buffer. When we output values from the fragment shader, we not only assign colors to fragments but also write the current depth value into the depth buffer. However, the depth buffer stores only the smallest z-value (frontmost) for each fragment position.

For instance, if we have a fragment at (300, 400) in the current framebuffer with a depth of 0.5 and receive a new fragment at the same location with a depth of 0.6, the new fragment, being further away, is discarded. Conversely, if a new fragment has a depth of 0.4, it's considered closer and replaces the old one. This ensures that the displayed fragments are never occluded by others.

The behavior of depth testing can be configured; we can choose to keep the fragment with the larger depth value and discard the smaller ones. However, in most cases, preserving the frontmost fragment is the preferred behavior.

Now, let's see how to update the code to enable depth testing. Fortunately, the shader code remains unchanged since writing to the depth buffer is automatically handled if the pipeline is configured to do so.

const pipelineDesc = {
    // ...
    depthStencil: {
        depthWriteEnabled: true,
        depthCompare: 'less',
        format: 'depth24plus-stencil8'
    }
};

In the pipeline descriptor, we introduce a new section called depthStencil. The depthWriteEnabled field is set to true, indicating that we want to write depth values to the depth buffer. For depthCompare, we specify 'less' to keep fragments with lesser depth values and discard those with greater depth. You can choose different behaviors, such as preserving fragments with greater depth.

Additionally, we specify the format as depth24plus-stencil8, allocating 24 bits for depth values and 8 bits for stencil values.

To complete our setup, we also need to create a depth buffer:

const depthTextureDesc = {
    size: [canvas.width, canvas.height, 1],
    dimension: '2d',
    format: 'depth24plus-stencil8',
    usage: GPUTextureUsage.RENDER_ATTACHMENT 
};

let depthTexture = device.createTexture(depthTextureDesc);
let depthTextureView = depthTexture.createView();

Creating the depth buffer is similar to creating a texture map. In the texture descriptor, we match the depth buffer's size to the canvas because each pixel on the canvas requires a corresponding depth value. The dimension is set to '2d', and we use the same depth24plus-stencil8 format.

We specify GPUTextureUsage.RENDER_ATTACHMENT as the usage because it serves as a rendering target, even though depth buffers are typically not visible.

const depthAttachment = {
    view: depthTextureView,
    depthClearValue: 1,
    depthLoadOp: 'clear',
    depthStoreOp: 'store',
    stencilClearValue: 0,
    stencilLoadOp: 'clear',
    stencilStoreOp: 'store'
};

const renderPassDesc = {
    colorAttachments: [colorAttachment],
    depthStencilAttachment: depthAttachment
};

Finally, we define the depthAttachment and the renderPass. Similar to the colorAttachment, the depthAttachment specifies the behavior when being loaded and when saving new values. We clear the depth buffer to 1 at the start (depthClearValue) as a depth value is only a scalar. During loading (depthLoadOp), we clear it to 1s to prepare for the rendering process, because 1 means the backmost depth in the Normalized Device Coordinate and during saving (depthStoreOp), we simply store the incoming value by the less behavior. Stencil settings can be ignored for now. In the renderPass, we now include a depthStencilAttachment.

With these updates, our code now correctly handles occlusion. Front surfaces occlude back surfaces, resulting in more accurate rendering.
