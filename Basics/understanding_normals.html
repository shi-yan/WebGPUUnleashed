1.15 Understanding Normals

In this tutorial, we explore the concept of normals, a fundamental aspect of understanding lighting in 3D scenes. Lighting effects in computer graphics depend on how light interacts with the material of a surface. The direction from which light strikes a surface can alter its appearance, making it crucial to determine where a surface is facing. This directional aspect is referred to as the normal direction.

To achieve flat surfaces, you can define a normal direction for each surface. For instance, on a cube, each side should possess a distinct normal direction. All points on the same surface should share the same normal direction, creating sharp transitions at edges and corners. This approach is suitable for objects with sharp creases, like a cube.

However, for more complex geometry, such as organic models like animals, we aim for smoother, more rounded surfaces instead of sharp creases. In such cases, defining normals on vertices is preferred.

Vertex normals are like averages of all the adjacent surface normals. When rendering the surface, interpolating vertex normals results in a smooth transition on the surface, achieving a polished look.

Normals can be calculated based on geometry or loaded from a file. Most 3D model files contain not just vertex positions but also normals. In this tutorial, we extend our loading logic to include normals.

    for (let v of obj.result.models[0].vertices) {
        positions.push(v.x);
        positions.push(v.y);
        positions.push(v.z);
        normals.push(0.0);
        normals.push(0.0);
        normals.push(0.0);
    }

    positions = new Float32Array(positions);
    normals = new Float32Array(normals);

    let positionBuffer = createGPUBuffer(device, positions, GPUBufferUsage.VERTEX);
    let normalBuffer = createGPUBuffer(device, normals, GPUBufferUsage.VERTEX);

It's not different from loading vertex positions. For each normal, it is also a vector of 3 float points. We just need to repeat what we've done for vertex positions in order to load normals.


@group(0) @binding(0)
var<uniform> modelView: mat4x4<f32>;
@group(0) @binding(1)
var<uniform> projection: mat4x4<f32>;
@group(0) @binding(2)
var<uniform> normal: mat4x4<f32>;

struct VertexOutput {
    @builtin(position) clip_position: vec4<f32>,
    @location(0) normal: vec3<f32>
};

@vertex
fn vs_main(
    @location(0) inPos: vec3<f32>,
    @location(1) inNormal: vec3<f32>
) -> VertexOutput {
    var out: VertexOutput;
    out.clip_position = projection * modelView * vec4<f32>(inPos, 1.0);
    out.normal = normalize(normal * vec4<f32>(inNormal, 0.0)).xyz;
    return out;
}

// Fragment shader

@fragment
fn fs_main(in: VertexOutput,  @builtin(front_facing) face: bool) -> @location(0) vec4<f32> {
    if (face) {
        var normal:vec3<f32> = normalize(in.normal);
        return vec4<f32>(normal ,1.0);
    }
    else {
        return vec4<f32>(0.0, 1.0, 0.0 ,1.0);
    }
}

Loading normals is similar to loading vertex positions. For each normal, represented as a vector of three floating-point numbers, we repeat the process used for vertex positions.

In the shader code, we introduce a new uniform called "normal transformation" because we need a distinct transformation for normals compared to the model-view matrix used for vertex positions. This separation is necessary for accurate rendering of normals.

In the vertex stage entry point, a new vertex attribute at @location(1) called inNormal is introduced. This attribute is a vector of three floats. We apply the normal transformation to this input. The output is usually passed to the fragment shader, where lighting calculations typically occur.

For this tutorial, we visualize normals by rendering them as colors. Normal vectors are normalized first since normal interpolation can affect their length. Normals are, by definition, unit vectors. This visualization helps us understand how normals are distributed on the surface before diving into lighting calculations.

    const normalAttribDesc = {
        shaderLocation: 1, // @location(1)
        offset: 0,
        format: 'float32x3'
    };

    const normalBufferLayoutDesc = {
        attributes: [normalAttribDesc],
        arrayStride: 4 * 3, // sizeof(float) * 3
        stepMode: 'vertex'
    };

In the JavaScript code, we introduce a normal attribute descriptor. It resembles the position attribute descriptor but specifies shaderLocation as 1 for normals.

    const pipelineDesc = {
        ...
        vertex: {
            module: shaderModule,
                entryPoint: 'vs_main',
                buffers: [positionBufferLayoutDesc, normalBufferLayoutDesc]
        },
        ...
    };

When defining the pipeline descriptor, both the position and normal buffers need to be provided for vertex buffers. During rendering, two vertex buffers must be set.

Now, let's render the teapot. You'll notice that the teapot's surface displays a rainbow of colors. These colors represent the normal directions at each point on the surface. Since our normals are defined at the vertex level, and GPU interpolation handles the transition from the vertex buffer to the fragment shader, the teapot's color transition is incredibly smooth. There are no visible triangles or sharp edges. This technique is instrumental in achieving realistic and smooth lighting effects.
