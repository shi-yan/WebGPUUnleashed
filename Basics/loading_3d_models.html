1.14 Loading 3D Models

In this tutorial, we will explore how to load 3D models from files. So far, we've been rendering simple geometries like planes and cubes, which are easy to define manually through vertex positions. However, in real-world 3D applications like games, we encounter much more complex geometries that cannot be crafted by hand. Therefore, we need to learn how to load these geometries from files.

The reason we covered index buffers in the previous tutorial is that using indices and a vertex pool is the common way of storing geometries in most 3D file formats. Therefore, using an index buffer is the most natural approach to rendering a geometry loaded from a file.

There are numerous 3D file formats, some of which are highly complex and often use binary encoding for storage efficiency. Covering such complex formats is beyond the scope of this tutorial. Instead, we will focus on one of the simplest file types, the .obj file. Despite its simplicity, the .obj format is widely used and has the advantage of being a human-readable textual format, making it helpful for debugging model loading and rendering. However, it lacks support for advanced features like lighting and animations, making it suitable primarily for storing geometries.

Here's a snippet of what an .obj file looks like:

v 0.000000 3.000000 1.800000
...
vn 0.2867 0.6914 -0.6631
...
f 2152//24 2158//24 2358//24

In general, an .obj file consists of consecutive data sections. The first section defines vertex positions, each starting with the letter v. Following this section, there may be optional sections for texture coordinates (prefixed with vt) and vertex normals (prefixed with vn). In this tutorial, we'll load a teapot model from a .obj file, which includes normals but no texture coordinates. We'll discuss normals in detail when covering lighting.

Next, there's a face section, defining triangles using vertex indices. Each line begins with the letter f, followed by indices referencing the vertex, texture coordinate, and normal sections, separated by slashes /. This structure aligns well with the index buffer and vertex buffer introduced earlier.

Rather than writing our own .obj file parser from scratch, we'll use an open-source .obj file parser called OBJFile.js. This parser generates a JavaScript object that allows us to query vertices and triangles.

Now, let's dive into the code. The shader code remains unchanged. In the JavaScript code, we use the fetch API to load the teapot .obj file. Since the .obj file is a human-readable text file, we obtain the raw data using response.text(). With the data loaded, we employ the OBJFile parser to obtain the parsed result.

    const objResponse = await fetch('teapot.obj');
    const objBody = await objResponse.text();

    let obj = await (async () => {
        return new Promise((resolve, reject) => {
            let obj = new OBJFile(objBody);
            obj.parse();
            resolve(obj);
        })
    })();

Parsing large models can be slow, so we use an asynchronous function. Once the .obj file is loaded, we need to assemble our position buffer. First, we query for the vertices. Since an .obj file can contain multiple 3D models, the models field is actually an array.

    let positions = [];

    for (let v of obj.result.models[0].vertices) {
        positions.push(v.x);
        positions.push(v.y);
        positions.push(v.z);
    }

    positions = new Float32Array(positions);

    let positionBuffer = createGPUBuffer(device, positions, GPUBufferUsage.VERTEX);

    let indices = [];

    for (let f of obj.result.models[0].faces) {
        for (let v of f.vertices) {
            indices.push(v.vertexIndex - 1)
        }
    }

    const indexSize = indices.length;

    indices = new Uint16Array(indices);

    const indexBuffer = createGPUBuffer(device, indices, GPUBufferUsage.INDEX);

Since the teapot.obj file contains only one model, we only query for model 0. We loop through the entire vertex pool, pushing all vertex locations into our positions buffer. Moving on to the index buffer, we loop through all faces and, for each face, loop through its three vertices, pushing the vertex indices into our indices array.

Notice that we decrement the vertex index by one. This is because, according to the .obj file format's definition, vertex indices should start from one, not zero. Thus, we subtract one when loading these indices. It's essential to remember that when loading the index buffer, we also need to obtain the index size, indicating how many indices need to be rendered.

With the index buffer populated, we create a GPU buffer as before. The rest of the code remains unchanged.

Now we have a more interesting geometry to work with. For those new to computer graphics, you might be surprised to learn that the humble teapot has been a ubiquitous presence in graphics experiments, often serving as a canvas to demonstrate various lighting techniques and other graphic innovations. Its history within the realm of computer graphics is quite intriguing. If you're curious to explore this further, you can find more details by following this link.
